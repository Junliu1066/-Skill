# Vibe Coding Kit

> 让一个**完全不懂技术、不懂产品的小白**，也能把一句模糊的想法，变成一套**专业级、能直接开发的交付物**——PRD + 高保真交互图 + 可施工的 SDD。

## 这是什么

一套给 [Claude Code](https://claude.com/claude-code) / Codex 用的 **Skill**。它的定位很特别：

- **对话照顾小白**：全程引导式选择题、大白话、讲清每个选择的代价、反复确认——零基础的人也能参与。
- **产出物专业级**：产出的 PRD / 交互图 / SDD，是按**资深产品经理 + 架构师**的标准来的，能直接支撑专业、生产级的开发。

一句话：**把小白嘴里的一句模糊话，拔高成专业三件套，去支撑做出一个能拿去市场验证的 demo。**

> 核心信念：「小白」只描述对话对象，不降低产出质量。范围可以小（先验证后商用），但工程纪律永远满级。

## 解决什么问题

非技术的人用 AI 写代码（Vibe Coding），最容易死在两件事上：

1. **需求没说清，AI 朝错误方向飞速实现** —— 本 Kit 用引导式问答 + 需求拆解 + 反复对齐来解决。
2. **产出太薄，AI 开不了工 / 越改越乱** —— 本 Kit 产出可施工的 SDD，并要求 AI 按一套「研发操作系统」（流水线 + 五角色 + 分区开发 + 红绿测试 + 安全/压力/评测 + 可观测/治理）来交付。

## 怎么用

1. 把 `skills/prd-helper/` 放进你的 Claude Code skills 目录。
2. 直接说你的想法，比如「我想做一个能帮我自动回复评论的东西」。
3. 跟着它的选择题一步步走，最后它会在一个文件夹里生成三件套。
4. 把三件套发给 Claude Code，让它照着开发。

## 它是怎么工作的（九步）

```
1. 问清场景 + 目标
2. 资格闸门      ← 竞品调研 + 理性分析，不值得做的诚实劝退，别盲目认同
3. 验证目标      ← 这 demo 拿去要验证什么、怎么算成功（本 Skill 的魂）
4. 拆解需求      ← 还原真实端到端流程，锁定核心能力与难点
5. 抠死处理规则  ← 把模糊动词抠成 AI 能执行的明确规则
6. 定技术方案    ← 形态 → 部署 → 技术栈，每个选项讲清代价
7. 补全空缺      ← 开发必需、小白想不到的细节用合理默认补满（绝不动核心）
8. 对齐到一致    ← 复述全貌，反复确认到「对，就是这样」才动手
9. 生成三件套    ← PRD + 高保真交互图 + 可施工 SDD
```

几条贯穿始终的硬规则：

- **不是每个想法都配往下走** —— 先理性筛一道。
- **简化可以砍周边，绝不能砍核心** —— 用户要验证的那个关键能力（如「自动」），再难也不能阉割；难/有风险就当场摊开、跟用户一起决定。
- **没拆清、没对齐，绝不动手**。
- **每个技术选择都讲代价** —— 不只是给答案，是教会用户「为什么」。

## 产出物：三件套

| 文件 | 管哪一层 | 标准 |
|------|---------|------|
| `1-PRD.md` | 产品层：做什么、规则是什么 | 专业产品经理水准的 PRD |
| `2-交互图.html` | 画面层：长什么样、怎么点 | 设计精良、可交互的高保真原型 |
| `3-开发文档.md` | 技术层：架构/模块/接口/任务/验收 | 架构师水准、能支撑生产级开发的 SDD |

> 合格判据：把 SDD 单独丢给一个没参与过对话的 AI，它能照着把东西建出来，且建出来的正是用户要的。

完整示例见 [`examples/小红书自动回复评论助手/`](examples/小红书自动回复评论助手)——一个真实跑出来的三件套（含运行记录）。打开 `2-交互图.html` 可直接看高保真原型。

## 路线图：现在是单 Skill，未来按生命周期扩成集群

**现在**：刻意保持**单 Skill**。需求对齐的九步是一场连续对话、强耦合，拆开会丢上下文、触发错乱；复杂度由 `references/` 渐进加载来管。先把它打磨到真有人用、真能产出好三件套。

**未来**（验证有人用之后）：沿「做产品」的生命周期扩成集群，各 Skill 用**三件套文件当接口**解耦——这正是「需求 → 开发 → 验证 → 治理」的全链路：

```
① 需求对齐 + 三件套生成   ← 本 Skill（已实现）
        ↓ 三件套
② SDD 驱动开发执行         ← 照 SDD 把东西真建出来（计划中）
        ↓ 代码
③ 验证 + 交付报告          ← 测试、证据、交付（计划中）
        ↓
④ 记忆 + 治理沉淀          ← ADR / 规则库 / 经验（计划中）
```

> 遵循本 Kit 自己的信条——**先验证、后扩张**：不在没验证需求前就摊开四个 Skill 的复杂基建。
