# 开发文档（SDD）—— 微信转账对账助手 v2

## 1. 概述

- **产品定位**：上传微信聊天截图，OCR 识别并提取转账信息，生成入金/出金对账表，支持编辑和导出 Excel
- **对应 PRD**：`1-PRD.md`
- **本期范围**：截图上传 → OCR 识别 → 提取出入金 → 分表展示 → 编辑 → 导出 Excel
- **本期不做**：用户登录、多人协作、支付集成

## 2. 总体架构

```
浏览器（用户设备）
  │ 上传截图 (PNG/JPG)
  ▼
Web 前端 (HTML/JS) ←─── Gin 后台 (Go)
  │                              │
  │ 展示、编辑、导出              │ 调百度云 OCR API
  │                              │
  ▼                              ▼
用户浏览器 ←─── PostgreSQL ──── 百度云 OCR API
```

## 3. 端到端流程的实现映射

| 步骤 | 用户/系统 | 实现模块 | 说明 |
|------|----------|---------|------|

## 6. 关键数据结构 / 接口约定

### 6.1 数据库表

```sql
CREATE TABLE records (
  id          SERIAL PRIMARY KEY,
  created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
  tx_time     VARCHAR(64) NOT NULL,           -- "2024-06-01 14:30"
  party       VARCHAR(128) NOT NULL,          -- 交易方昵称
  direction   VARCHAR(8) NOT NULL CHECK (direction IN ('入金', '出金')),
  amount      DECIMAL(12,2) NOT NULL CHECK (amount >= 0.01 AND amount <= 999999.99),
  note        TEXT DEFAULT '',
  source_file VARCHAR(256) DEFAULT '',
  dedup_hash  VARCHAR(64) UNIQUE,             -- SHA256(tx_time+party+amount)
  updated_at  TIMESTAMP NOT NULL DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_records_direction ON records(direction);
CREATE INDEX idx_records_tx_time ON records(tx_time DESC);
CREATE INDEX idx_records_direction_time ON records(direction, tx_time DESC);
```

**去重策略**：`INSERT ... ON CONFLICT (dedup_hash) DO NOTHING`，通过 `cmdTag.RowsAffected()` 判断是否重复。

### 6.2 REST API

**统一错误格式**：
```json
{ "error": "描述信息", "code": 400 }
```

#### POST /api/upload
上传截图并识别。

请求：`multipart/form-data`，字段 `file`（图片二进制，限制 PNG/JPG，≤10MB）

响应 200：
```json
{
  "records": [
    {
      "id": 1,
      "tx_time": "2024-06-01 14:30",
      "party": "张三",
      "direction": "入金",
      "amount": 200.00,
      "note": "",
      "source_file": "screenshot_001.png"
    }
  ],
  "total": 5,
  "duplicated": 0
}
```

响应 400（格式不对）：
```json
{ "error": "仅支持 PNG 和 JPG 格式", "code": 400 }
```

#### GET /api/records?direction=in
获取记录列表。

参数：`direction` — `in` 或 `out`

响应 200：
```json
{
  "records": [
    { "id": 1, "tx_time": "2024-06-01 14:30", "party": "张三", "direction": "入金", "amount": 200.00, "note": "" }
  ],
  "total": 5
}
```

#### PUT /api/records/:id
编辑单条记录。

请求体：
```json
{
  "tx_time": "2024-06-01 14:30",
  "party": "张三",
  "direction": "入金",
  "amount": 200.00,
  "note": ""
}
```

响应 200：
```json
{ "record": { "id": 1, "tx_time": "2024-06-01 14:30", ... } }
```

响应 422（校验失败）：
```json
{ "error": "金额必须大于 0", "code": 422 }
```

#### DELETE /api/records/:id
删除记录。

响应 200：`{ "ok": true }`

#### GET /api/export?direction=in
导出 Excel。

参数：`direction` — `in` 或 `out`（省略则导出全部，两个 sheet）

响应：`Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`，文件名 `转账对账_入金_2026-06.xlsx`

### 6.3 核心算法：OCR 文本解析引擎 ★

这是整个系统最关键的模块（`parser/parser.go`）。以下完整描述从 OCR 原始文本到结构化记录的转换过程。

#### 输入示例（OCR 返回的原始 JSON）

```json
{
  "words_result": [
    { "words": "2024年6月1日 14:30" },
    { "words": "张三" },
    { "words": "向你转账¥200.00" },
    { "words": "收款成功" },
    { "words": "" },
    { "words": "2024年6月1日 14:35" },
    { "words": "李四" },
    { "words": "向你转账¥580.00" },
    { "words": "收款成功" }
  ]
}
```

（百度云 OCR 接口中 `words_result` 是按行返回的数组）

#### 算法步骤（伪代码）

```
function ParseRecords(words_result):
    records = []
    current_block = []
    
    for each item in words_result:
        text = item.words
        
        // 遇到空行或时间戳行 → 闭合当前块
        if text == "" or matches_time_pattern(text):
            if current_block not empty:
                record = parse_block(current_block)
                if record != null:
                    records.append(record)
            current_block = []  // 开始新块
        
        current_block.append(text)
    
    // 处理最后一个块
    if current_block not empty:
        record = parse_block(current_block)
        if record != null:
            records.append(record)
    
    return records
```

```
function parse_block(lines):
    text = join(lines, "\n")
    
    // 1. 排除非转账
    if contains(text, "系统通知") or contains(text, "已退还") 
       or contains(text, "已过期") or (contains(text, "红包") and not contains(text, "转账")):
        return null
    
    // 2. 判定方向
    if matches(text, "向你转账|收到转账|已收款|收款¥|入账¥"):
        direction = "入金"
    else if matches(text, "你向.*转账|转出¥|付款¥|支出¥"):
        direction = "出金"
    else:
        return null  // 非转账记录
    
    // 3. 提取金额
    amount_match = regex_find(text, "[¥￥](\d{1,3}(,\d{3})*\.\d{2})")
    if amount_match:
        amount_str = amount_match[1].replace(",", "")
        amount = parse_float(amount_str)
    else:
        amount_match = regex_find(text, "(\d+\.\d{2})元")
        if amount_match:
            amount = parse_float(amount_match[1])
        else:
            return null  // 无合法金额，跳过
    
    // 4. 提取时间（取块中第一个匹配的行）
    time_match = regex_find(text, "\d{4}[年-]\d{1,2}[月-]\d{1,2}[日\s]+\d{1,2}:\d{2}")
    tx_time = normalize_time(time_match)  // 转为 "2024-06-01 14:30"
    
    // 5. 提取交易方（取时间行后的第一个非空非关键词行）
    party = extract_party(lines, time_line_index, direction)
    // 简单策略：取时间行之后的第1-2行，跳过"向你转账"等关键词行
    
    // 6. 生成去重哈希
    dedup_hash = SHA256(tx_time + "|" + party + "|" + format_amount(amount))
    
    return Record(tx_time, party, direction, amount, "", dedup_hash)
```

#### 正则模式表

| 目标 | 正则 | 示例匹配 |
|------|------|---------|
| 时间 | `\d{4}[年-]\d{1,2}[月-]\d{1,2}[日\s]+\d{1,2}:\d{2}` | `2024年6月1日 14:30` / `2024-06-01 14:30` |
| 入金关键词 | `向你转账\|收到转账\|已收款\|收款[¥￥]\|入账[¥￥]` | `向你转账¥200.00` |
| 出金关键词 | `你向.+转账\|转出[¥￥]\|付款[¥￥]\|支出[¥￥]` | `你向王五转账¥1,200.00` |
| 金额（¥前缀） | `[¥￥](\d{1,3}(,\d{3})*\.\d{2})` | `¥1,200.00` → 1200.00 |
| 金额（元后缀） | `(\d+\.\d{2})元` | `200.00元` → 200.00 |

#### 测试用例

| 用例 | 输入文本片段 | 期望输出 |
|------|-------------|---------|
| 正常入金 | `"2024年6月1日 14:30\n张三\n向你转账¥200.00\n收款成功"` | `{tx_time:"2024-06-01 14:30", party:"张三", direction:"入金", amount:200.00}` |
| 正常出金 | `"2024年6月3日 09:15\n你向王五转账¥1,200.00\n输入支付密码"` | `{tx_time:"2024-06-03 09:15", party:"王五", direction:"出金", amount:1200.00}` |

| 缺交易方 | `"2024年6月4日\n向你转账¥50.00"`（无昵称行） | `{tx_time:"2024-06-04 00:00", party:"未知", direction:"入金", amount:50.00}`（兜底） |

## 7. 技术选型（已定）

| 层面 | 选型 | 理由 |
|------|------|------|
| 后台 | Go + Gin | 单文件部署、内存小、CRUD 场景成熟 |
| 前端 | 纯 HTML + CSS + JS | 最小依赖，内嵌在 Go 二进制中 |
| 数据库 | PostgreSQL | 支持并发、适合未来扩展 |
| OCR | 百度云 OCR（通用文字识别） | 价格低，免费额度够用，有 Go SDK |
| Excel | excelize | Go 最成熟的 Excel 库 |
| 部署 | Railway/Fly.io | 自动 HTTPS、PostgreSQL 托管、部署简单 |

**不选其他方案的理由**：
- Python/Flask：依赖管理复杂，部署不如 Go 省心
- SQLite：并发写入有限，未来迁移成本高

## 9. 安全底线

- 密钥仅通过环境变量传递，不写代码、不进前端
- 上传文件类型校验：仅允许 PNG/JPG，MIME 校验 + 魔数校验
- 上传文件大小限制：≤ 10MB
- API 输入校验：direction 仅接受 `入金`/`出金`，amount 限定范围
- 使用参数化查询（database/sql 的 PreparedStatement）
- 错误信息不泄露内部细节，统一返回用户能看懂的中文提示
- HTTPS 由托管平台自动配置
