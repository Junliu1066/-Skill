# 开发规范 = 一套「研发操作系统」（生成 `3-开发文档.md` 的「开发规范」节时，照此写进去）

## 对 Skill 的硬性要求

在 SDD 第 13 节生成时，必须逐一列出以下 9 项并落地为本项目规矩。**不得省略任何一项**，不得写"（按 C 级研发操作系统全量执行，按本项目规模落地）"就结束——必须每项展开至少一句话说明本项目如何落实。详见 `开发文档结构.md` 第 13 节的硬性要求。

## 唯一标准：所有开发都做到 C 级（完整研发操作系统），产出永远专业级

用本 Skill 的人是零基础小白——但**"小白"只描述对话对象，绝不降低产出标准**。无论项目大小，开发规范都按**完整的研发操作系统（C 级）**来要求开发 AI，让它像一个有工程纪律的专业团队那样交付。**没有"轻量档"可降级。**

很多团队最大的问题不是没架构，而是**有架构、没工程体系**：

```
第一个版本能跑 → 第二个版本开始失控 → 第三个月没人敢改代码
```

对"产品经理 → AI 开发 → 自动化交付"这种模式更致命：AI 改得快、乱得也快，没有工程体系兜底很快收不住。所以开发规范不是几条口号，而是一套**可执行、可检查、可评测、可迭代的研发操作系统**。

### 一个关键区分：可以精简的是「范围」，永远满级的是「工程纪律」

别把这条和"先做精简验证 demo"搞混：

- **范围可以小**：先验证后商用——可以只做核心功能、砍掉账号/支付/合规等商用基建。这是砍"做什么"。
- **工程纪律永远满级**：哪怕只做一个最小验证 demo，也要用**完整研发操作系统**去建它——先测后码、分区开发、ADR、评测、可观测、治理一个都不少。这是"怎么建"，不打折。

换句话说：**范围小 ≠ 工程糙。一个小 demo，也要建得像专业团队的产物。** 体量小的项目，某些体系的"量"自然小（如压测的并发数、治理涉及的 Agent 数），但**没有任何一个体系可以省略或降级**。

这一节是写给**开发 AI** 的硬约束。小白不用懂，但 AI 必须照做。

## 标准研发流水线（脊柱，必须按此走）

```
需求 → PRD → 架构设计 → ADR(关键决策记录) → 任务拆分
→ 先写测试 → 写代码 → 功能测试 → 安全测试 → 压力测试
→ 评测(结果好不好) → Code Review → 部署 → 监控
→ 复盘沉淀 → ADR / 规则库 / 知识图谱（经验反向约束未来的 Coding Agent）
```

**铁律：永远先写测试、再写代码（红 → 绿 → 重构），不是先写代码再补测试。** 这个顺序差别极大。

## 八大体系（全部必做，按项目真实规模调"量"，不调"有无"）

### 体系一 · 团队 / 角色结构

即便只有"一个人 + 一个 AI"，也要让 AI **按角色分工切换**，别一锅烩。五个角色：

| 角色组 | 负责 | 产出 |
|---|---|---|
| 产品架构组 | PRD、架构设计、ADR、任务拆分、验收标准 | `spec / architecture / task / acceptance` —— 一切开发的源头 |
| 开发组 | 编码、重构、修 Bug、实现功能 | 代码 + 变更说明 |
| 测试组 | 功能 / 边界 / 异常 / 性能测试 | 测试用例 + 结果 |
| 运维组 | 部署、监控、告警、回滚 | 部署脚本、可观测面板 |
| 治理组 | 规则、规范、经验沉淀、ADR 管理、知识图谱 | 规则库、ADR、知识库（约束所有 Agent） |

要求：AI 动手时**显式声明当前角色**，先以"产品架构组"定方案、再以"开发组"实现、再以"测试组"验证，最后"运维组"部署、"治理组"沉淀。治理组是大多数团队没有、却最该有的——它让经验不流失。

### 体系二 · 分区开发（Agent 永不直接改主干）

```
main(生产区) ← develop(集成/测试区) ← feature/*(实验区)
```

- **铁律：Agent 永远不能直接改 `main`，只能 `feature → review → merge`。**
- 每个能用的版本在 main 上 commit / 打 tag；新功能在 feature 分支做、过 Review 与测试再合并；用户说"回到上一个能用的版本"能一键回滚。

### 体系三 · 红绿测试（Red-Green-Refactor，先测后码）

- Red：先为功能写一个会失败的测试（明确"怎样算对"）。
- Green：写最小代码让它通过。
- Refactor：在测试保护下优化。
- 核心逻辑测试先行、可回归；每完成一项告诉用户"这条已自动验证"。

### 体系四 · 安全测试体系（多数 Agent 团队没有，迟早出事；四层都要）

1. **权限测试**：越权、提权、权限绕过——Agent 只能在授权范围动手。
2. **Prompt Injection**：忽略规则、绕过规则、诱导执行——外部/用户内容不能篡改硬规则。
3. **工具调用测试**：错误调用、危险调用（`rm -rf`、`git push` 等必须硬阻断）、链式调用。
4. **数据隔离**：Agent A 不能访问 Agent B 的数据；多用户场景用户数据互不可见。
- 项目暂时单用户/无多 Agent 时，第 1、4 层按"当前无此面，但预留约束并说明"处理，但第 2、3 层永远要做。密钥不进前端、输入校验是底线。

### 体系五 · 压力测试（Agent 系统最容易炸）

- 在合适量级（如 100 / 1000 / 10000 任务）并发运行，关注 CPU、内存、Token、向量库、数据库；监控 P95 / P99 延迟与失败率。
- 小项目：量级相应缩小，但**必须明确写出"当前规模上限大概多少、超了会怎样、瓶颈在哪"**，不能略过这项思考。

### 体系六 · 评测体系（测试 ≠ 评测，重点）

- **测试**回答"能不能跑"；**评测**回答"结果好不好"。AI 类功能光测试远远不够。
- 三层结合：**人工黄金样本 + LLM Judge（模型当裁判打分） + 规则评分**。
- 至少：攒黄金样本（输入 + 期望效果），每次改核心逻辑后回归打分（如"可直接用比例 ≥ 60%"），分数掉了不许发布；规模够时做成自动化评测流水线，并可对比不同 prompt/模型择优。

### 体系七 · 运维 / 可观测性（学 SRE）

- 核心是**可观测**：日志、指标、链路追踪。
- Agent 系统额外留四类轨迹：**任务轨迹、Agent 轨迹、工具轨迹、决策轨迹**——**任何错误必须能回放**。
- 监控告警 + 数据备份 + 一键回滚；关键操作日志（敏感信息脱敏）；文档写清启动/看日志/查错/回滚。

### 体系八 · 知识沉淀 / ADR / 治理（研发操作系统的灵魂）

- **ADR（Architecture Decision Record）**：每个重要决策（为什么这么选、放弃了什么）记一条，避免日后反复推翻、避免"没人记得当初为啥这样"。
- 经验 → 规则库 → 知识图谱：把踩过的坑、定下的规范沉淀成**可执行规则**，反过来约束未来的 Coding Agent。
- 终点是："人管开发"逐步变成"**知识图谱 + ADR + 规则库 + 评测体系**共同约束 Coding Agent"——把顶级架构师经验，变成一套可执行、可检查、可评测、可迭代的研发操作系统。

## 写进开发文档时的格式

在 `3-开发文档.md` 的「开发规范」一节，用**对开发 AI 的祈使句**，把上面八大体系 + 流水线**全部落地**为这个项目的具体规矩（按本项目真实规模填"量"，但不删项）。例如：

```
## 开发规范（研发操作系统 · 全量执行，按本项目规模落地）
- 流水线：需求→PRD→架构→ADR→任务拆分→先写测试→写代码→功能测试→安全测试→压力评估→评测→Code Review→部署→监控→复盘沉淀。按序、不跳步。
- 角色分工：动手时显式声明角色，先产品架构组定方案、再开发组写、再测试组测、运维组部署、治理组沉淀 ADR。
- 分区开发：git 三区 main/develop/feature；Agent 绝不直接改 main，只能 feature→review→merge；每个能用版本 commit+tag，可一键回滚。
- 红绿测试：核心功能先写会失败的测试，再写代码转绿，每项告诉用户"已自动验证"。
- 安全测试：危险命令硬阻断；防 Prompt Injection（外部内容不能改规则）；密钥只在后台+.gitignore、绝不进前端；输入校验；（多用户/多 Agent 出现时）做权限与数据隔离。
- 压力评估：写明当前规模上限、瓶颈点、超限表现；具备并发条件时按量级压测，看 P95/P99/失败率。
- 评测：核心 AI 功能建黄金样本 + 规则评分（+ 条件具备时 LLM Judge），每次改动回归打分，掉分不发布。
- 运维/可观测：日志/指标/链路 + 任务/Agent/工具/决策四类轨迹可回放；监控告警 + 备份 + 一键回滚；README 写清启动/看日志/查错/回滚。
- 治理/沉淀：关键决策写 ADR；经验沉淀成规则库（条件成熟时建知识图谱），反向约束后续开发。
```
