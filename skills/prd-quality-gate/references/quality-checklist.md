# PRD 质量检查清单

此文件由 `prd-quality-gate` 技能使用，提供结构化的质量检查项。

---

## 一、PRD 强制性检查清单

### 1. 粒度要求（硬性）

- [ ] **功能模块（Fxx）** ≥ 3 个（不满足 → 拒收，必须补全）
- [ ] **用户故事（USxxx）** ≥ 5 个（不满足 → 拒收，必须补全）
- [ ] **验收标准（ACxx）** ≥ 5 个（不满足 → 拒收，必须补全）
- [ ] 每个功能模块下至少包含 1 个用户故事
- [ ] 每个用户故事至少包含 1 条验收标准

### 2. 格式规范（硬性）

- [ ] 文档层级清晰：版本号 → 变更记录 → 功能模块（Fxx）→ 用户故事（USxxx）→ 验收标准（ACxx）
- [ ] 所有模块使用 `### Fxx - 模块名称` 格式
- [ ] 所有用户故事使用 `#### USxxx - 故事标题` 格式
- [ ] 所有验收标准使用 `- **ACxx**：` 格式（列表项，编号+描述）
- [ ] 编号连续，无跳号
- [ ] 标题明确：用户故事标题需概括核心价值，验收标准标题需体现可测试性

### 3. 内容完整性（硬性）

- [ ] 每个用户故事包含：作为...我想要...以便... 的标准格式
- [ ] 每个验收标准包含明确的输入、预期输出/行为
- [ ] 涉及 UI 的验收标准附带页面草图/线框图（可用 ASCII art 替代）
- [ ] 涉及数据交互的验收标准附带接口时序图（可用 mermaid 格式）

### 4. 可测试性（建议）

- [ ] 每条验收标准可以用 1-3 个测试用例覆盖
- [ ] 验收标准不含歧义描述（如"系统应快速响应" → 改为"响应时间 < 200ms"）
- [ ] 异常情况和边界条件已覆盖

## 二、SDD 强制性检查清单

### 1. 粒度要求（硬性）

- [ ] **模块设计** ≥ 3 个（不满足 → 拒收，必须补全）
- [ ] **接口定义** ≥ 5 个（不满足 → 拒收，必须补全）
- [ ] **数据库表设计** ≥ 3 张表（不满足 → 拒收，必须补全）

### 2. 格式规范（硬性）

- [ ] 文档层级清晰：架构概览 → 模块设计 → 接口定义 → 数据库设计
- [ ] 接口定义包含：方法、路径、请求参数、响应格式、错误码
- [ ] 数据库表设计包含：表名、字段名、类型、约束、索引、说明
- [ ] 编号连续，无跳号

### 3. 内容完整性（硬性）

- [ ] 每个模块包含职责说明、依赖关系、对外接口
- [ ] 每个接口包含完整的请求/响应示例
- [ ] 每张表包含字段列表和索引设计
- [ ] 包含技术栈选型说明和理由

### 4. 技术合理性（建议）

- [ ] 技术选型有明确的理由（性能、团队能力、生态等维度）
- [ ] 接口设计遵循 RESTful 规范（如适用）
- [ ] 数据库设计满足第三范式（如适用）
- [ ] 考虑了安全性和性能优化

## 三、HTML 原型强制性检查清单

### 1. 完整性（硬性）

- [ ] 所有 PRD 中定义的核心页面均已生成 HTML 原型
- [ ] 所有交互状态已覆盖（默认、悬停、激活、禁用、加载、空状态、错误状态）

### 2. 规范性（硬性）

- [ ] 使用纯 HTML + CSS + vanilla JS，不依赖任何框架或库
- [ ] 所有 CSS 使用内联样式或 `<style>` 标签，无外部依赖
- [ ] 原型可在浏览器中直接打开，无需构建工具

### 3. 交互性（建议）

- [ ] 主要用户流程可点击操作
- [ ] 包含示例数据，便于理解产品形态
- [ ] 响应式设计（如适用）
