# 开发文档（SDD）—— 微信转账对账助手

## 1. 概述

- **产品定位**：上传微信聊天截图，自动识别并提取转账信息，生成入金/出金两张对账表，支持编辑和导出 Excel
- **对应 PRD**：`1-PRD.md`
- **本期范围**：截图上传 → OCR 识别 → 提取出入金 → 分表展示 → 手动编辑 → 导出 Excel
- **本期不做（non-goals）**：用户登录/账号系统、多人协作、支付集成、在线分享

## 2. 总体架构

```
浏览器（用户设备）
  │ 上传截图 (PNG/JPG)
  ▼
Web 前端 (HTML/JS) ←─── Gin 后台 (Go)
  │                              │
  │ 展示结果、编辑、导出           │ 调用百度云 OCR API
  │                              │
  ▼                              ▼
用户浏览器 ←─── PostgreSQL 数据库 ──── 百度云 OCR
```

- **前端**：纯 HTML + CSS + JS（无框架，单页面应用），通过 HTTP API 与后台通信
- **后台**：Go + Gin 框架，处理文件上传、调用 OCR、数据管理
- **数据库**：PostgreSQL，存储识别记录
- **外部依赖**：百度云 OCR API（文字识别）

## 3. 端到端流程的实现映射

| 用户流程步骤 | 由谁实现 | 说明 |
|-------------|---------|------|
| ① 打开页面 | 前端 HTML | 加载页面骨架，显示上传区域和示例引导 |
| ② 上传截图 | 前端上传组件 → POST /api/upload | 用户选择图片文件，传给后台 |
| ③ OCR 识别 | 后台 `/api/upload` → 调百度云 OCR | 后台接收图片，调用 OCR 识别文字 |
| ④ 提取出入金 | 后台 OCR 结果解析模块 | 按规则从识别文本中提取转账记录 |
| ⑤ 去重检查 | 后台 OCR 结果解析模块 | 基于时间+交易方+金额去重 |
| ⑥ 存储到 DB | 后台数据库模块 | 写入 PostgreSQL |
| ⑦ 返回结果 | 后台 → GET /api/records | 返回 JSON 供前端渲染 |
| ⑧ 分表展示 | 前端渲染模块 | 按入金/出金分两张表展示 |
| ⑨ 手动编辑 | 前端编辑框 → PUT /api/records/:id | 用户点击编辑，提交修改 |
| ⑩ 导出 Excel | 后台 → GET /api/export | 生成 Excel 文件下载 |

## 6. 关键数据结构 / 接口约定

### 数据模型（Record）

```sql
CREATE TABLE records (
  id          SERIAL PRIMARY KEY,
  created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
  tx_time     VARCHAR(64) NOT NULL,         -- 转账时间，如 "2026-06-01 14:30"
  party       VARCHAR(128) NOT NULL,        -- 交易方
  direction   VARCHAR(8) NOT NULL,          -- '入金' 或 '出金'
  amount      DECIMAL(12,2) NOT NULL,       -- 金额
  note        TEXT DEFAULT '',              -- 备注
  source_file VARCHAR(256) DEFAULT '',      -- 来源截图文件名（用于追溯）
  dedup_hash  VARCHAR(64) UNIQUE            -- 去重哈希：tx_time+party+amount
);
```

### API 接口

| 方法 | 路径 | 输入 | 输出 | 说明 |
|------|------|------|------|------|
| POST | `/api/upload` | multipart/form-data: file | `{ records: [...] }` | 上传截图，返回识别结果 |
| GET | `/api/records?direction=in` | query: direction | `{ records: [...] }` | 获取入金/出金记录 |
| PUT | `/api/records/:id` | `{ tx_time, party, direction, amount, note }` | `{ record: {...} }` | 编辑记录 |
| DELETE | `/api/records/:id` | - | `{ ok: true }` | 删除记录 |
| GET | `/api/export` | query: direction | Excel 文件流 | 导出为 Excel |

## 7. 技术选型（已定）

| 层面 | 选型 | 理由 |
|------|------|------|
| 后台语言 | Go | 单文件部署、内存小、适合 CRUD+API 场景 |
| 后台框架 | Gin | Go 最主流的 HTTP 框架，资料多 |
| 前端 | 纯 HTML + CSS + JS | 不引入框架，减少依赖，直接内嵌在 Go 二进制中 |
| 数据库 | PostgreSQL | 支持并发、适合未来扩展为多用户 |
| OCR | 百度云 OCR | 价格便宜，免费额度高，有 Go SDK |
| Excel 导出 | excelize | Go 最成熟的 Excel 库 |
| 部署 | 托管平台（Railway / Fly.io 等） | 自动 HTTPS、数据库托管、部署简单 |

**不选的其他方案**：
- Python/Flask：依赖管理复杂，部署不如 Go 省心
- Node.js：对零基础用户来说 Go 的二进制部署更友好
- SQLite：并发写入有限，未来迁移成本高
- MySQL：功能重叠，PostgreSQL 对 JSON/全文检索支持更好

## 8. 跑在哪 / 部署

### 部署步骤（托管平台）

请带用户一步步做，每步讲清为什么、并告诉他怎么确认这步成功：

1. **选平台**：推荐 Railway 或 Fly.io（对新手友好，有免费/便宜额度，月费约 ¥5-20）
2. **注册账号**：带用户注册，需要的话说明要不要绑卡
3. **上传代码**：教用户把代码推送到 GitHub，平台自动拉取部署
4. **配密钥**：在平台后台配置环境变量：
   - `BAIDU_OCR_API_KEY` —— 百度云 OCR 的 API Key
   - `BAIDU_OCR_SECRET_KEY` —— 百度云 OCR 的 Secret Key
   - `DATABASE_URL` —— PostgreSQL 连接串（平台会自动提供）
5. **配数据库**：在平台后台创建 PostgreSQL 数据库实例，平台自动生成 `DATABASE_URL`
6. **部署上线**：点部署按钮，等平台编译 Go 项目并启动
7. **验证**：用平台给的网址在手机和电脑各打开一次、各上传一张截图试试，正常＝成功
8. **费用提醒**：告诉用户在哪看用量和账单，避免不知不觉超额

⚠️ 调百度云 OCR 每次约 0.001-0.003 元，免费额度通常够个人用户使用。

## 9. 安全底线

- **密钥不入前端**：百度云 API Key/Secret Key 只在后台环境变量中使用，绝不暴露给前端
- **输入校验**：上传文件限制为 PNG/JPG，大小 ≤ 10MB
- **SQL 注入防护**：使用参数化查询（GORM 或原生 prepared statement）
- **不出错信息**：对外 API 不暴露内部错误堆栈，返回用户能看懂的中文提示
- **HTTPS**：托管平台自动配置 HTTPS

## 12. 测试计划

### 功能测试
- 上传一张包含 5 条以上转账记录的清晰截图 → 识别出正确数量的记录
- 上传不包含金额的普通聊天截图 → 返回"未识别到转账记录"
- 上传非图片文件 → 返回"请选择 PNG 或 JPG 格式的图片"

### 边界测试
- 上传同一张截图两次 → 第二次全部标记为重复，不新增记录
- 编辑记录后刷新页面 → 编辑的内容持久化
- 删除记录后刷新 → 记录消失

### 异常测试
- 百度云 OCR 鉴权失败 → 后台日志记录错误，前端提示"OCR 服务异常，请检查密钥"
- 数据库连接断开 → 后台返回 503，前端提示"服务暂时不可用"
- 上传超大图片（>10MB）→ 前端拦截提示

## 13. 开发规范

### 研发操作系统（全量执行，按本项目规模落地）

- **流水线**：架构设计 → 任务拆分 → 先写测试 → 写代码 → 功能测试 → 部署 → 复盘沉淀。按序、不跳步。
- **角色分工**：动手时显式声明角色，先以"产品架构组"定方案、再以"开发组"实现、再以"测试组"验证、最后"运维组"部署。
- **分区开发**：main 分支始终保持可部署状态。每个 Task 建 feature 分支，完成后合并到 main。
- **红绿测试**：核心逻辑（OCR 结果解析引擎、去重逻辑）先写测试用例，再实现代码。每完成一项验证测试通过。
- **安全测试**：
  - 密钥只通过环境变量传递，不写死、不进前端
  - 图片上传做文件类型和大小校验
  - API 输入做参数校验（防 SQL 注入）
- **压力评估**：本项目单用户使用，不考虑高并发。近期不需要压测。
- **评测**：OCR 识别效果用 10 张以上不同截图测试，记录准确率。每次改解析规则后回归。
- **运维/可观测**：后台日志记录关键操作（上传、识别、导出）。README 写清启动/看日志/查错/回滚。
- **治理/沉淀**：开发中踩到的坑做成 README 的 FAQ 节，供后续参考。
