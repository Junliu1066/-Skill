> 这是 architecture（S2 选型）和 prd（S1.8 技术栈雏形）的**内容来源**。
>
> **🗓 离线优先 · 盘点日期：2026-06　·　有效期：约一年（到 2027 年中前可信）**
> 选型时**直接读这份 md，不需要联网**——这样在 Claude Code / ChatGPT / Codex 任何环境都能用、且快。
> 唯一要留意的是过期：超过有效期、或你听说某个栈有重大变化时，再按文末「如何更新这份库」联网刷新一次即可。

## 怎么用（顺序不能反）

```
① 先定产品形态（它活在哪）          ← prd S1.3 已经定好
        ↓
② 按形态找到下面对应的平台分类
        ↓
③ 取该平台的 ★ 默认栈作为推荐项      ← 最成熟、AI 写得最好、最好维护
        ↓
④ 拿「现有资源清单」逐项对照调整      ← 有现成服务就用，没服务器就砍后端
        ↓
⑤ 给用户讲清代价，让他点头或换一个    ← 推荐 + 代价 + 可否决
```

**三条铁律（贯穿所有平台）：**

1. **先问"要不要"，再问"用什么"。** 很多项目根本不需要后端 / 不需要原生 App——能用纯前端、纯静态、小程序解决的，别上重家伙。
2. **优先用你已经有的。** 已有飞书多维表、Supabase、某云账号、静态托管——优先拿它当地基，别让 AI 新起一套。
3. **默认推"最成熟"，不是"最新潮"。** 越主流的栈，AI 训练数据越多、生成质量越高、出错越好查。把语言交给 AI，但形态和"该用哪个平台方案"由这张库帮你定。

## 网站

| 场景 | ★ 推荐默认栈 | 何时选 | 代价 / 提醒 |
|------|------------|--------|------------|
| 纯展示 / 官网 / 留资 / 工具小页（无后端） | **单 HTML 文件** 或 **静态站（HTML+CSS+JS）**，扔 GitHub Pages / Vercel / 对象存储 | 内容不常变、不需登录、不存数据 | 最省心、几乎零成本；"表单提交存哪"先用本地记录或发邮件，后端列入以后再说 |
| 有交互 / 要登录 / 动态内容 / 要 SEO | **Next.js（React）** + Tailwind CSS | 要做成正经产品、页面多、要被搜索引擎收录 | 复杂度跳一档，要会部署；但 AI 对 React 训练最多，生成质量最高 |
| 熟悉 Vue / 想要平缓学习曲线 | **Nuxt（Vue）** + Tailwind | 团队/你更习惯 Vue 的模板写法 | 生态略小于 React，AI 生成质量仍很好 |
| 只想最快出一个能点的原型 | **Vite + React/Vue** | 验证想法、还没想做成正式站 | 默认无 SEO/SSR，纯前端单页 |

> **默认建议**：拿不准就先做**静态站**。"显得正经"不该是上框架的理由——成熟官网很多就是静态页 + 表单。

## 小程序

| 场景 | ★ 推荐默认栈 | 何时选 | 代价 / 提醒 |
|------|------------|--------|------------|
| **只做微信小程序** | **微信原生开发** | 单一平台、要性能、坑最少 | 只能跑微信；但最稳、性能最好 |
| 要同时上多端（微信 + H5 + 其它小程序） | **uni-app**（熟 Vue） | 想一套代码多端复用、团队上手快 | 跨端有约 10–20% 性能损耗、包体积偏大 |
| 多端 + 想和 Web 端共享业务逻辑 | **Taro（React）** | 熟 React、Web 和小程序共用一套逻辑 | 运行时方案，高频交互场景性能损耗更明显 |

> **选择原则**：只做微信 → 认真考虑原生；要跨端，熟 Vue 选 uni-app、熟 React 选 Taro；都不熟 → uni-app 门槛更低。

## 移动端（手机 App）

> **先问一句：真的需要原生 App 吗？** 很多需求用 **PWA（网页加到桌面）** 或 **小程序** 就够了，省掉应用商店审核、签名、长期上架维护这一整摊事。确认真要 App 再往下看。

| 场景 | ★ 推荐默认栈 | 何时选 | 代价 / 提醒 |
|------|------------|--------|------------|
| 跨平台 App（iOS + Android 一套代码） | **Expo（React Native）** | 想从零最快跑起来、熟 JS | 官方推荐的 RN 起步方式，不用装 Xcode/Android Studio；MVP 和快速迭代最强 |
| UI 重、想要漂亮且一致的界面 | **Flutter** | 界面复杂、想"像请了设计师" | 一套 SDK、自带组件多；但用 Dart 语言，和 Web 技能不通用 |
| 其实只是想"装到桌面" | **PWA**（给网站加 manifest） | 不需要上架、不需要原生能力 | 最省事；但通知/硬件能力受限 |

> 原生 App 的隐性成本最高：应用商店审核、开发者账号年费、签名证书、版本上架。能不上 App 就先别上。

## 后端接口

> **同样先问：真的需要自建后端吗？** 纯前端 + 一个现成 BaaS（后端即服务）往往就够，能省掉"养一台服务器"的全部负担。

| 场景 | ★ 推荐默认栈 | 何时选 | 代价 / 提醒 |
|------|------------|--------|------------|
| **已有现成服务**（飞书多维表 / 企业微信 / 某 BaaS） | **直接用它当后端** | 你手里已经有 | 零新增运维；优先级最高，别让 AI 另起炉灶 |
| 要数据库 + 登录 + 存储，又不想运维 | **Supabase**（PostgreSQL + 认证 + 存储 + Edge Functions） | 要正经数据层、又是一个人维护 | 有官方 MCP、对 AI 协作友好；Pro 档约 $25/月，有厂商绑定 |
| 想完全自己掌控、要最轻 | **PocketBase**（单文件二进制 + SQLite + 自带后台） | 自托管、预算敏感、规模不大 | 免费，只付服务器钱；扛不住大规模 |
| 要写自定义接口逻辑、熟 Python | **FastAPI** | 业务逻辑复杂、需要自己写接口 | 要管一台常开服务器，它挂服务就停 |
| 要写自定义逻辑、熟 JS / 想免运维 | **云函数 / Serverless**（Vercel、腾讯云云函数等） | 逻辑不重、按调用计费 | 冷启动延迟；按量计费要留意成本 |

> **代价对照**：自建后端（FastAPI/Node + 自己的服务器）= 要管机器、要运维、挂了服务就停；BaaS（Supabase/PocketBase/云函数）= 省运维，但有订阅费或厂商绑定。**一个人维护的项目，优先 BaaS 或现成服务。**

## 拿"现有资源清单"反推（贴合资源，别脱离现实）

| 你的现状（资源清单里） | 直接砍掉 / 优先选 |
|----------------------|------------------|
| 没服务器、不想买 | 砍掉一切要常开后端的方案；网站走静态托管，后端走 BaaS/云函数 |
| 已有静态托管（Vercel / GitHub Pages / 对象存储） | 网站优先纯前端，扔上去即可 |
| 已有某现成服务（飞书多维表 / Supabase 等） | 直接当后端，别新建 |
| 有 AI 接口 key、对成本敏感 | 让 AI 估每次调用成本，把"省调用"写进方案 |
| 不会写代码 + 想长期自己维护 | 选最主流、依赖最少、出错好查的；能少一个组件就少一个 |

## 选定后，写进项目说明书的「选定技术栈」

照下面格式落盘（与 `harness.json` 对「选定技术栈」的要求一致：必须含**选定**与**否决**）：

```
## 选定技术栈
- 产品形态：{纯前端网页 / 小程序 / 移动端 App / …}
- 选定：{如 静态站(HTML+JS) + 表单先发邮件}
- 理由：{贴合哪条现有资源 + 为什么最好维护}
- 否决：{如 Next.js——这版无需登录无需 SEO，上框架是过度复杂}
- 成本提醒：{如 暂无按量费用 / Supabase 约 $25 月}
```

## 如何更新这份库（不必常做）

平时**不用联网**，直接用就行。只在两种情况下刷新一次：① 过了顶部「有效期」；② 你听说某平台技术有大变化（如某框架被废弃、某 BaaS 大幅调价）。

刷新做法（交给 AI）：
> "联网核对一下 `references/技术栈推荐库.md` 里网站 / 小程序 / 移动端 / 后端接口四类的主流成熟栈，在 {当前年份} 是否还成立。有变化的，更新对应行 + 顶部盘点日期；没变化的，只把日期往后挪。"

> 想省心可以把它设成**定时任务**：每半年提醒刷新一次。
