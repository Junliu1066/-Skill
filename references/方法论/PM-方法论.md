# 产品经理 — 方法论

## PRD 写作框架

### 1. 问题陈述优先（先问题，后方案）

在给出任何解决方案之前，先定义：
- **谁有这个问题？**（用画像描述，不是人口统计）
- **他们现在怎么解决的？**（当前的临时办法 / workaround）
- **现在的方案为什么不够好？**（痛点强度 1–10 分）
- **我们有什么证据？**（用户访谈、数据、客服工单）

### 2. 需求优先级排序

用 **RICE 模型**给功能打分：

| 因子 | 定义 | 量纲 |
|---|---|---|
| **R**each（触达） | 每季度影响多少用户？ | 实际数字 |
| **I**mpact（影响） | 对核心指标的推动有多大？ | 3=巨大, 2=高, 1=中, 0.5=低, 0.25=极小 |
| **C**onfidence（信心） | 我们有多确定？ | 100%=高, 80%=中, 50%=低 |
| **E**ffort（成本） | 需要多少人月？ | 实际估算 |

**RICE 分 =（触达 × 影响 × 信心）÷ 成本**

### 3. 验收标准规范（EARS）

每条验收标准都必须：
- **具体**——不用含糊词（"快"、"好用"、"简单"）
- **可测**——有数字或可二元判断
- **独立**——不依赖其他验收标准就能测
- **格式**——**当【前置条件】，在【动作】时，则【可观察的结果】**

❌ 差："页面应该加载得快"
✅ 好："当用户处于 3G 网络，打开 /dashboard 时，首次内容绘制（FCP）应 < 2 秒"

❌ 差："登录应该安全"
✅ 好："当同一 IP 在 10 分钟内登录失败 5 次，第 6 次尝试时，返回 429 并锁定 15 分钟"

### 4. 边界情况清单

每个功能都必须考虑：
- 空状态（还没有数据）
- 错误状态（接口失败、校验失败）
- 加载状态（进行中）
- 边界值（0、1、最大值、负数）
- 并发用户（竞态条件）
- 离线 / 慢网络
- 权限不足
- 已删除 / 过期数据

### 5. 非功能需求模板

| 类别 | 需求 | 目标 | 如何度量 |
|---|---|---|---|
| 性能 | 页面加载时间 | FCP < 1.5s | Lighthouse CI |
| 性能 | 接口响应时间 | p95 < 200ms | 服务端指标 |
| 性能 | 并发用户 | 同时 1000 | 压力测试 |
| 安全 | 鉴权 | 带刷新的 JWT | 人工审计 |
| 安全 | 数据加密 | TLS 1.3 + 静态 AES-256 | 安全扫描 |
| 安全 | 输入校验 | 所有接口 | 自动化测试 |
| 可访问性 | WCAG 等级 | 2.1 AA | axe-core 审计 |
| 可访问性 | 键盘导航 | 所有可交互元素 | 人工测试 |
| 可靠性 | 可用性 | 99.9% | 监控 |
| 可靠性 | 错误率 | < 0.1% | 错误追踪 |

### 6. 成功指标框架（HEART）

- **H**appiness（愉悦）——用户满意度（NPS、CSAT）
- **E**ngagement（参与）——使用频率、会话时长
- **A**doption（采用）——新用户、功能采用率
- **R**etention（留存）——次日 / 7 日 / 30 日留存
- **T**ask success（任务成功）——完成率、完成耗时

每个指标都要有：**基线 → 目标 → 度量方法 → 复盘频率**。

### 7. PRD 常见错误（要避开）

1. **方案先于问题**——还没定义用户需求就跳到"我们需要一个按钮"
2. **验收标准含糊**——"应该很直观"无法测试
3. **漏掉边界情况**——只有正常路径，没有错误处理
4. **没有成功指标**——上线了却不知道有没有成功
5. **范围蔓延内置**——"如果还能……就好了"却没标成"范围外"
6. **缺非功能需求**——没有性能目标、没有安全要求
7. **没有用户流程**——只有功能清单，不展示它们怎么串联
8. **预设实现**——在 PRD 里写"用 React"（那是架构，不是产品）

> 对君留：本文件直接作为"需求拆解 Skill"的核心骨架——尤其【问题优先】【EARS 验收】【边界清单】【非功能模板】【HEART 指标】这五块，正好填进需求规格的对应字段。
