# 开发文档规范

> 本规范用于把 PRD 和项目说明书转成 AI 能直接开发的 SDD（Software Design Document）。
> PRD 负责回答“做什么、为什么做、验收什么”；项目说明书负责回答“技术栈、约束、部署、安全”；开发文档负责回答“具体怎么实现、按什么接口、按什么任务开发、怎么测试”。

---

## 1. 文档定位

开发文档不是需求说明，也不是泛泛的技术方案。

它的目标是：让 AI 或开发者在不反复追问的情况下，能按模块、接口、数据结构、任务和测试计划逐步实现产品。

合格标准：

- AI 不需要猜目录结构、接口字段、错误格式、状态流转。
- AI 知道每个模块应该放在哪个文件、承担什么职责。
- AI 知道先做哪一步、每一步完成后怎么验收。
- AI 遇到矛盾、缺决策、做不出来时，会暂停并回填「决策复盘 / 案例库」。
- 用户能按验收清单逐条手动验证。

---

## 2. 文件位置

| 产出物 | 路径 | 模板 |
|--------|------|------|
| 开发文档 | `docs/开发文档.md` | `examples/开发文档-模板.md` |

规则：

- 正式项目必须写入 `docs/开发文档.md`。
- 不得把开发文档混进 PRD。
- 不得只写“使用某技术栈实现”，必须写到接口、数据、模块、任务和测试层。

---

## 3. 必须包含的章节

### 3.1 概述

必须写清：

- 产品定位
- 对应 PRD
- 对应项目说明书
- 本期范围
- 本期不做
- 项目元信息：语言、框架、端口、配置方式、运行方式

### 3.2 总体架构

必须写清：

- 前端、后端、数据库、外部服务之间的关系
- 是否同域部署
- 是否本地运行
- 是否依赖第三方 API
- 数据是否出本机

推荐用简单架构图，不追求复杂。

### 3.3 端到端流程

必须按用户真实操作写完整链路：

| 步骤 | 角色 | 实现位置 | 说明 |
|------|------|----------|------|
| 1 | User | 页面 / 命令 / 文件 | 用户做什么 |
| 2 | System | 模块 / 函数 / API | 系统怎么处理 |
| 3 | System | 数据库 / 文件 / 外部服务 | 数据怎么流转 |

要求：

- 每一步都要能映射到代码位置。
- 不能只写“处理数据”“生成结果”这种空话。

### 3.4 模块清单

必须列出每个模块：

| 模块 | 路径 | 职责 | 不负责 |
|------|------|------|--------|

要求：

- 一个模块只承担清晰职责。
- 明确“不负责什么”，避免 AI 把逻辑塞错地方。

### 3.5 目录结构

必须给出目标目录树。

要求：

- 每个核心文件后面写一句职责说明。
- 不要生成大而空的目录。
- 小项目优先少目录、少层级。

### 3.6 数据结构与存储

根据项目形态填写：

- 前端状态结构
- 后端模型
- 数据库 DDL
- 本地文件格式
- 缓存结构
- 去重规则
- 索引与约束

要求：

- 字段名、类型、是否必填、默认值必须明确。
- 有数据库时必须写建表语句或迁移规则。
- 有本地文件时必须写文件格式示例。

### 3.7 接口约定

有 API、CLI、文件输入输出、组件 props 时必须写。

每个接口至少包含：

- 名称 / 路径 / 命令
- 请求参数
- 响应格式
- 成功示例
- 失败示例
- 校验规则
- 错误提示

错误格式必须统一，不能每个接口一套。

### 3.8 核心算法 / 业务规则

凡是 AI 容易猜错的规则，必须写成伪代码或规则表。

适合写在这里的内容：

- 解析规则
- 分类规则
- 状态流转
- 金额、时间、文件名、权限等处理规则
- 排除规则
- 优先级规则
- 去重规则

要求：

- 不只写“智能识别”“自动分类”。
- 必须写输入、处理步骤、输出。
- 边界情况必须写。

### 3.9 外部服务与依赖

涉及第三方服务时必须写：

- 服务名称
- 调用方式
- 是否使用 SDK
- 鉴权方式
- 环境变量名
- 限流
- 超时
- 重试
- 错误处理
- 降级方案

禁止写入真实密钥、密码、token。

### 3.10 基础设施与配置

必须写清：

- 环境变量
- `.env.example`
- 端口
- 日志格式
- 超时设置
- 文件大小限制
- 跨域策略
- 数据库迁移方式
- 健康检查

### 3.11 安全底线

至少覆盖：

- 密钥不进代码、不进前端
- 输入校验
- 文件类型校验
- 权限边界
- 参数化查询
- 错误信息不泄露内部细节
- 用户隐私数据处理
- 删除 / 覆盖 / 支付等高风险操作的确认机制

### 3.12 分阶段里程碑

必须按能跑通的最小链路拆：

| 阶段 | 目标 | 完成标准 |
|------|------|----------|
| M0 | 最小闭环 | 能完成主链路 |
| M1 | 核心功能补齐 | 核心 CRUD / 状态 / 展示可用 |
| M2 | 导出 / 部署 / 增强 | 可交付 |

要求：

- M0 必须是能运行的闭环，不是搭空架子。
- 每个阶段都有可观察完成标准。

### 3.13 开发任务总表

必须拆到 AI 能逐项执行：

| Task ID | 阶段 | 任务 | 验收标准 |
|---------|------|------|----------|

要求：

- Task ID 稳定，如 `T-M0-01`。
- 每个任务有明确输入和完成标准。
- 不要出现“完善系统”“优化体验”这种任务。

### 3.14 测试计划

至少包含：

- 单元测试
- API / 集成测试
- 端到端测试
- 边界测试
- 错误输入测试
- 手动验收脚本

核心规则必须给具体测试用例：

- 输入是什么
- 期望输出是什么
- 如何判断通过

### 3.15 给开发 AI 的执行规则

必须写清：

- 按里程碑顺序开发，先跑通 M0。
- 每完成一个任务，对照验收标准自检。
- 遇到矛盾、缺决策、做不出来、需要偏离开发文档时，必须暂停。
- 用户提出新想法时，先记入「以后再说」或「决策复盘 / 案例库」，不要直接插入当前开发。
- 如果用户明确同意偏离，必须记录偏离原因和代价。

### 3.16 反馈回路

必须有固定格式：

```md
## 反馈回路

### 必须暂停的情况
- 需求技术上无法实现
- 两个需求互相矛盾
- 当前开发文档缺少关键决策
- 实现结果和用户预期明显不一致
- 继续开发会引入明显安全、成本或维护风险

### 处理步骤
1. 用大白话说明卡点
2. 给出 2-3 个选项和各自代价
3. 等用户确认
4. 更新开发文档
5. 把本次案例写入「决策复盘 / 案例库」
```

---

## 4. 不合格示例

以下写法不合格：

- “前端实现上传页面，后端处理文件。”
- “数据库保存用户数据。”
- “接口返回成功或失败。”
- “系统正常运行。”
- “AI 自动识别内容。”
- “后续优化体验。”

原因：这些话不能指导开发，也不能验收。

---

## 5. 合格示例

合格写法应该像这样：

```md
POST /api/upload
请求：multipart/form-data，字段 file，支持 PNG/JPG，最大 10MB
成功响应：{"records":[...],"total":5,"duplicated":0}
失败响应：{"error":"仅支持 PNG 和 JPG 格式","code":400}
校验：同时检查 MIME 和文件头魔数
验收：上传 .txt 文件时返回 400，页面显示“仅支持 PNG 和 JPG 格式”
```

这类写法能直接变成代码、测试和验收动作。

- [ ] 已写入 `docs/开发文档.md`
- [ ] 概述、本期范围、本期不做已明确
- [ ] 架构、流程、模块、目录结构已明确
- [ ] 数据结构、接口、错误格式已明确
- [ ] 核心算法 / 业务规则写到可实现程度
- [ ] 外部服务、配置、安全底线已明确
- [ ] 里程碑和任务总表可逐项执行
- [ ] 测试计划包含具体输入和期望输出
- [ ] 反馈回路包含“暂停、选项、确认、更新文档、沉淀案例”
- [ ] 无密钥、密码、token 等敏感信息
- [ ] 正文为简体中文
