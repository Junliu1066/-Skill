# Harness v2 重设计方案（评审稿）

> 文档状态：草稿（待评审）　·　版本：v0.1　·　目标：把 Harness 从"产出物检查员"升级成"流程管理员"

---

## 一、要解决的问题

测试中发现：**纯 Skill 在对话里不严格按预设步骤推进**——AI 会乱序、跳步、需求还没问完就冲去写代码。

根因不在 skill 写得不好，而在 Harness 的**定位**：

| 现状 | 问题 |
|------|------|
| `harness.json` 只定义"产出什么文件、文件里有哪些章节、字段格式" | 全是**名词（产出物）**，管不到**动词（过程）** |
| harness skill 只在 AI 声称"完成"后校验文件 | 校验发生得**太晚**，过程中跑偏全程无反馈 |
| SKILL.md 里的步骤是建议性剧本 | 步骤**不是门**，AI 可乱序、跳步，没人拦 |
| 没有"当前在第几步"的记录 | AI **不知道自己走到哪**，长对话里更会忘 |

一句话：**当前 Harness 管"交出来的东西对不对"，但不管"做的过程对不对"。**

---

## 二、设计目标与约束

- **力度：账本级中等约束。** 不依赖 Claude Code 专属 hook（那只在一个平台生效）。靠"一个进度账本文件 + AI 每轮必须读写它"的仪式来约束。ChatGPT / Codex / Claude Code 通用。
- **颗粒度：步骤级门。** 不只在 skill 之间设门，连 skill 内部的每一步都有"退出条件"，过了才能进下一步。
- **不破坏现有脾气。** 深度自适应（分诊）仍在：轻需求允许跳步，但**跳步必须在账本里留痕**——把"随意乱跳"变成"有意识地、留记录地跳"。
- **三层单源派生不变。** 仍然 `harness.json` 是唯一真实来源，`CLAUDE.md` 和 harness skill 从它派生。

---

## 三、核心机制：进度账本（Progress Ledger）

新增一个文件 `docs/进度账本.md`，作为整个流程的**唯一状态来源**。

### 3.1 它是什么

- AI **每轮对话开场第一件事**：读这个账本，向用户报"当前在 S2·选型 的第 3 步"。
- AI **每做完一步**：更新账本（标记该步已完成、当前步骤 +1）。
- 它同时解决"AI 忘事"：长对话或开新对话时，把账本贴回去，AI 立刻知道走到哪。

### 3.2 账本格式（固定结构，便于人读和 AI 维护）

```markdown
# 进度账本

- 项目名称：卡密自动发货中间层
- 当前阶段：S1 · 需求对齐
- 当前步骤：S1.4（资源清单已盘，正在定形态）
- 最近更新：2026-06-26

## 阶段进度

| 阶段 | 状态 | 出口门 | 备注 |
|------|------|--------|------|
| S1 需求对齐 | 进行中 | ✗ 未过 | 走 prd 访谈路径 |
| S2 架构选型 | 未开始 | — | 被 S1 阻塞 |
| S3 上线准备 | 未开始 | — | demo 阶段暂不进入 |

## 当前阶段的步骤

| 步骤 | 必经? | 状态 | 退出条件是否满足 |
|------|-------|------|------------------|
| S1.0 破冰 | 必经 | ✅ 完成 | 拿到一句话 + 谁用 |
| S1.1 分诊定深度 | 必经 | ✅ 完成 | 判定为"正式系统"，走深问 |
| S1.2 资源清单 | 必经 | ✅ 完成 | 清单已回填 |
| S1.3 定形态 | 必经 | 🔄 进行中 | 待用户确认推荐项 |
| S1.4 补全需求 | 必经 | ⬜ 未开始 | — |

## 跳步留痕（深度自适应）
- （示例）S1.4 信息架构：已跳过。理由：无界面的纯脚本项目，不适用。

## 阻塞项
- 无
```

> **为什么用 Markdown 而不是 JSON：** 用户会把 skill 贴进 ChatGPT/Codex，AI 维护 Markdown 表格比维护严格 JSON 更稳、更不易格式崩。机器校验由 harness skill 读这张表完成。

## 五、每个 SKILL.md 的改造模板

每个 skill 从"建议剧本"改成"带门的状态机"，统一加三段。**这是改动量最大的部分，但结构高度一致，可批量套用。**

```markdown
## 准入检查（开始本 skill 前必做）

1. 读 `docs/进度账本.md`。若不存在，创建它并初始化到本阶段第 0 步。
2. 确认上一阶段 exit_gate 已过（看账本）。未过 → 告诉用户"需要先回到 SX 补完 YY"，不要开始。
3. 向用户报告："当前在 S1·需求对齐，准备进入第 N 步。"

## 步骤（每一步 = 一道门）

### 步骤 S1.3 · 定形态
- 做什么：[原有内容……给选项+推荐+代价]
- **退出条件**：「它活在哪」已选定一项（用户确认了推荐项或自选）。
- **过门动作**：在账本里把 S1.3 标记 ✅，当前步骤改为 S1.4。条件没满足就停在本步，别往下走。
- （若分诊判为"轻"且本步 required=false：可跳，但在账本"跳步留痕"写一行理由。）

…（其余步骤同构）…

## 出口门（声称本 skill 完成前必做）

- [ ] 本阶段所有 required 步骤已 ✅（看账本）
- [ ] 产出物通过 harness.json 里本 skill 的 field_requirements / validation
- 全过 → 账本里本阶段标记「已完成、出口门✓」→ 提示用户"可进入下一阶段 SX"。
- 任一项没过 → 不准声称完成，继续补。
```

## 六、harness skill 升级：从检查员到审计员

现有 harness skill 保留所有产出物检查，**新增一节"流程一致性审计"**（读账本核对过程）：

```markdown
### 3.0 流程一致性检查（新增，最先做）
读 docs/进度账本.md，核对：
- 阶段是否按序推进（没有跳过整个阶段而未留痕）
- 已标「完成」的阶段，其 exit_gate 是否真的过了（防止"账本说完成、产出物没达标"的假完成）
- required 步骤有没有被偷偷跳过（跳步必须有留痕+理由）
- 当前账本状态与 docs/ 下实际产出物是否吻合
报告：🔴 流程违规（如"S1未过却已在S2")／🟡 跳步未留痕／✅ 流程一致
```

并在质检报告里新增一块"流程审计结果"，和现有"产出物检查""质量门"并列。

## 十、轻模式：合并报门（避免 demo 被问得太啰嗦）

步骤级门默认是"重模式"（逐步停顿、逐步确认）。当**分诊（S1.1）判定为"轻"**（自己用 / 一次性 / 跑 demo）时，自动切到**轻模式**：

- **必经步骤照做，但允许"几步合并、一次性报门"**，不必每步都停下来等用户确认。
- **`required:false` 步骤默认跳过**，在账本"跳步留痕"里一行带过（如"S1.5 信息架构：纯脚本，跳"）。
- **合并分组（prd 为例）**：
  - 组 A = S1.0 破冰 + S1.1 分诊 + S1.2 资源清单 →（一次性把"做什么/谁用/有什么资源"问清，报一次门）
  - 组 B = S1.3 定形态 + S1.4 补全 + S1.6 边界 →（推荐项 + 快速扫缺口 + 一句边界，报一次门）
  - 组 C = S1.7 验收 + S1.9 落盘 →（给可验证验收 + 直接落盘）
- **报门动作**：每组结束在账本里把该组所有步骤一并标 ✅，向用户复述一次"我已确认 A 组：你要做 X、自己用、本机脚本——对吗？"得到点头即过。
- **硬底线不打折**：无论轻重，**出口门（exit_gate）一律要过**——也就是 prd.md / 项目说明书.md 该有的核心字段不能因为"轻"而缺。轻模式省的是"过程中的停顿次数"，不是"产出质量"。

> 重模式（给别人用 / 正式系统）则严格逐步报门，一步一确认。模式由账本里 S1.1 的分诊结论驱动，全程可被用户一句话改回（"我们正式点，逐步来"）。
